# Security Policy

## Supported Versions

| Version | Supported |
| --- | --- |
| `[latest major or branch]` | Yes |
| `[older version]` | No |

If you are unsure whether your version is supported, include the version number in your report.

## Reporting a Vulnerability

Please do not report suspected security vulnerabilities in public issues, discussions, or pull requests.

Report vulnerabilities privately through:

- [Security email]
- [GitHub Security Advisory link, if enabled]
- [Private intake form, if used]

Include as much of the following as possible:

- Affected version or commit.
- Description of the vulnerability.
- Steps to reproduce.
- Proof-of-concept code, logs, screenshots, or traces.
- Potential impact.
- Known mitigations or workarounds.
- Whether the issue is already public.
- Preferred name or handle for credit, if you want public credit.

## Response Expectations

Expected process:

1. Maintainers acknowledge the report within [time window].
2. Maintainers investigate impact and affected versions.
3. Maintainers coordinate a fix or mitigation.
4. Maintainers publish patched versions when appropriate.
5. Maintainers disclose details after users have a reasonable upgrade path.

Do not promise a fixed timeline unless the maintainer team can consistently meet it.

## Disclosure Policy

We ask reporters to give maintainers a reasonable opportunity to investigate and fix the issue before public disclosure.

Maintainers will not ask reporters to hide risk from users indefinitely. If a fix is delayed, maintainers should communicate status and coordinate a responsible disclosure plan.

## Security Updates

Security fixes may be released as:

- Patch releases.
- Backported fixes for supported versions.
- Configuration guidance.
- Documentation updates.
- Temporary mitigation instructions.

Release notes or advisories will describe user action required when it is safe to publish that information.

## Out of Scope

The following are usually out of scope unless they demonstrate a concrete exploit path:

- Reports against unsupported versions.
- Missing security headers in documentation-only websites.
- Social engineering.
- Denial-of-service reports without practical impact.
- Vulnerabilities in third-party services outside project control.
- Automated scanner output without analysis.

## Safe Harbor

Security research conducted in good faith is welcome when it:

- Avoids privacy violations.
- Avoids data destruction.
- Avoids service disruption.
- Uses only the access needed to verify the issue.
- Reports findings privately and promptly.

Good-faith researchers who follow this policy should not be penalized by the project for reporting vulnerabilities responsibly.
